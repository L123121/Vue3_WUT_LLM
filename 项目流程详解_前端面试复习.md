# 武理小精灵：项目流程详解

> 适用场景：前端面试复习、项目讲解、简历项目深挖  
> 阅读目标：就算没系统学过前端，也能看懂这个项目是怎么跑起来的

## 1. 先用一句话理解这个项目

这是一个 **AI 聊天助手** 应用。  
你登录后，可以：

1. 和 AI 助手聊天（支持流式输出）
2. 管理多个聊天会话
3. 导入 GitHub Skills 自定义 AI 行为
4. 切换主题、语言

前端负责"页面和交互"，后端负责"提供接口和 AI 服务"。

---

## 2. 项目里最重要的几个概念（新手必看）

### 2.1 前端是什么？

前端就是你在浏览器里看到的页面和操作逻辑。  
本项目前端技术：

1. **Vue 3**：写页面组件，使用 Composition API
2. **Pinia**：管理全局状态（用户信息、聊天记录、Skills）
3. **Vue Router**：页面跳转（`/login`、`/chat`）
4. **Tailwind CSS**：快速写样式
5. **Vite**：开发和打包工具

### 2.2 后端是什么？

后端是运行在服务器上的程序，主要提供 API 接口。  
本项目后端是 Node.js + Express，负责：

1. 登录接口
2. 聊天接口（流式返回）
3. 对接讯飞/Qwen 模型（失败时可走 mock）

### 2.3 "状态"是什么？

状态就是"程序当前记住的数据"。  
比如：

1. 当前登录用户是谁
2. 当前是深色模式还是浅色模式
3. 当前聊天会话是哪一个
4. 每个会话有哪些消息
5. 已导入哪些 Skills

这些状态放在 Pinia 里，页面只负责显示和触发动作。

## 4. 整个项目"从启动到可用"的完整流程

### 4.1 启动阶段

1. 运行前端：`npm run dev`
2. 运行后端：`cd backend && npm run dev`
3. 浏览器打开前端地址（默认 5173）

此时 Vite 会把前端里的 `/api` 请求转发到后端 `http://localhost:3000`。

### 4.2 前端初始化发生了什么？

入口文件：`src/main.js`

它做三件事：

1. 创建 Vue 应用
2. 挂载 Pinia（状态管理）
3. 挂载 Router（路由）

然后挂到页面根节点 `#app`。

### 4.3 全局壳层如何工作？

核心文件：`src/App.vue`

它负责：

1. 如果当前是 `/login`，只渲染登录页
2. 如果不是 `/login`，渲染完整后台布局：
   - 左侧 Sidebar（会话列表）
   - 顶部 Header（标题、用户头像）
   - 中间主内容区（路由页面）
3. 挂全局组件：`ToastManager`（全局提示）

你可以把它理解成"网站外壳"。

## 5. 登录流程（一步一步）

页面：`src/views/Login.vue`  
状态：`src/stores/auth.store.js`  
路由守卫：`src/router/index.js`

### 5.1 用户行为

1. 输入账号和密码
2. 点击登录按钮

### 5.2 前端逻辑

1. 先做空值校验
2. 模拟请求延迟
3. 密码为 `123456` 时视为成功
4. 调 `authStore.login(...)` 写入用户信息
5. 显示成功 toast
6. 跳转聊天页 `/chat`

### 5.3 路由守卫逻辑

1. 访问受保护页面且未登录 -> 跳到 `/login`
2. 已登录还访问 `/login` -> 重定向 `/chat`

这就是"前端权限门禁"。

## 6. AI 聊天全链路（最重要）

这一段是面试核心，建议重点背。

### 6.1 相关文件

1. 页面容器：`src/views/AIChat.vue`
2. 会话状态：`src/stores/chat.store.js`
3. API 封装：`src/api/chat.js`
4. 会话列表：`src/components/chat/ConversationList.vue`
5. 消息列表：`src/components/chat/MessageList.vue`
6. 输入框：`src/components/chat/ChatBox.vue`
7. 语音输入：`src/components/chat/VoiceRecorder.vue`
8. Markdown 渲染：`src/components/chat/MarkdownRenderer.vue`
9. Skills 面板：`src/components/chat/SkillPanel.vue`

### 6.2 用户发送一条消息后发生什么？

#### 第 1 步：输入

用户在 `ChatBox` 输入文字，按 Enter 或点发送。

#### 第 2 步：触发 store 发送

`AIChat.vue` 调 `chatStore.sendMessage(message)`。

#### 第 3 步：store 先改本地状态

在 `chat.store.js` 里：

1. 先把用户消息加入当前会话
2. 创建一个空的 AI 消息占位（用于流式追加）
3. `isLoading = true`
4. 记录当前流式消息 id：`currentStreamingId`

这样 UI 能立刻反馈，不用等接口返回。

#### 第 4 步：构建历史上下文

`buildHistory(...)` 会做清洗：

1. 去掉 welcome 消息
2. 去掉错误消息
3. 取最近 20 条
4. 修正连续同角色问题

目的是提高模型上下文质量，减少噪声。

#### 第 5 步：构建系统提示（Skills）

如果有启用的 Skills，`skillStore.buildSystemPrompt()` 会：

1. 取最多 4 个启用的 Skill
2. 组装成系统提示消息
3. 插入到对话历史前面

#### 第 6 步：调用流式接口

通过 `sendMessageStream(...)` 请求 `/api/stream`。

接口是 SSE 风格，服务端会一段一段返回：

1. 收到 chunk -> `onChunk` -> 追加到 AI 占位消息文本
2. 收到 `[DONE]` -> `onDone` -> 结束 loading
3. 出错 -> `onError` -> 显示错误文案并结束 loading

#### 第 7 步：页面实时刷新

`MessageList.vue` 监听消息变化自动滚动到底部。  
用户看到的效果就是"AI 一边生成一边显示"。

## 7. 会话管理机制（为什么这是亮点）

`chat.store.js` 不是"单数组消息"，而是"多会话模型"：

1. `conversations`：所有会话
2. `currentConversationId`：当前会话 id
3. `currentConversation`：当前会话对象（computed）
4. `messages`：当前会话消息列表（computed）

支持功能：

1. 新建会话
2. 切换会话
3. 重命名会话
4. 删除会话
5. 会话列表按更新时间排序
6. 首条用户消息自动生成标题

这套结构很接近真实 AI 产品的前端架构。

## 8. Skills 功能（新增亮点）

### 8.1 什么是 Skills？

Skills 是一种"自定义 AI 行为"的机制。用户可以从 GitHub 导入 SKILL.md 文件，让 AI 按照特定风格或规则回答问题。

### 8.2 相关文件

- `src/stores/skill.store.js`：Skills 状态管理
- `src/components/chat/SkillPanel.vue`：Skills 面板 UI

### 8.3 导入流程

1. 用户粘贴 GitHub blob 链接（如 `https://github.com/xxx/blob/main/SKILL.md`）
2. 前端将链接转换为 raw 链接（`https://raw.githubusercontent.com/...`）
3. 通过 fetch 获取 Markdown 内容
4. 解析标题和描述，存入 localStorage
5. 发送消息时，将启用的 Skills 组装成系统提示

### 8.4 技术细节

- **URL 转换**：支持 `github.com/blob/...` 和 `raw.githubusercontent.com/...` 两种格式
- **去重**：相同 raw URL 的 Skill 不会重复导入
- **限制**：最多 4 个 Skill 同时启用，每个 Skill 内容限制 6000 字符

## 9. 本地持久化机制（刷新不丢数据）

本项目把核心状态存进了 localStorage：

1. 聊天会话：`chat_conversations`
2. 当前会话 id：`chat_current_conversation_id`
3. Skills：`chat_skills`
4. 主题：`darkMode`
5. 语言：`app_language`

另外还做了旧数据迁移：

1. 检测旧 key `chat_messages`
2. 自动转换成新会话结构
3. 删除旧 key

这体现了"版本升级兼容意识"。

## 10. 国际化和主题机制

### 10.1 国际化

`language.store.js + i18n/messages.js`

1. 维护 `zh-CN / en-US` 词典
2. 提供 `t(path)` 和 `tm(path)` 读取文案
3. 支持参数插值，如 `{count}`

### 10.2 深色模式

`theme.store.js`

1. `darkMode` 状态变化时，给 `document.documentElement` 加/删 `dark` 类
2. 同步写入 localStorage
3. 页面刷新仍保留主题

## 15. 大厂面试常问问题（附答题方向）

### Q1：为什么聊天状态要放 store？

因为聊天数据跨多个组件共享，而且生命周期长（切路由、刷新恢复），放组件会导致数据分裂和难恢复。

### Q2：为什么要先插入空 AI 消息？

为了稳定渲染目标，后续 chunk 只做字符串追加，UI 更平滑，逻辑也更可控。

### Q3：为什么要做历史裁剪？

避免上下文太长带来性能和成本问题，也减少模型被无关内容干扰。

### Q4：Skills 功能是怎么实现的？

1. 用户输入 GitHub 链接
2. 转换成 raw URL 并 fetch 内容
3. 解析 Markdown 提取标题和指令
4. 存入 localStorage
5. 发消息时组装成系统提示

### Q5：这个项目离生产还差什么？

1. 真正鉴权（JWT/刷新机制）
2. 聊天历史后端持久化
3. 更完整测试（单测/集成/e2e）
4. 监控埋点与错误追踪
5. 配置安全（敏感密钥必须只放环境变量）

## 18. 面试官追问脚本版（问题 + 高分回答）

> 使用方法：先读"短答版"应对快问，再用"展开版"应对深挖。

### Q1：你这个项目最核心的技术难点是什么？

短答版：
核心难点是 AI 聊天链路的状态管理，我需要同时处理多会话、流式输出、Skills 提示、错误恢复和刷新持久化，保证体验连续性。

展开版：
项目不是只把接口结果显示出来，而是要做"连续对话产品体验"。我把聊天状态拆成 `conversations/currentConversationId/isLoading/currentStreamingId`，发送时先落用户消息和 AI 占位消息，再按 SSE chunk 增量更新同一条 AI 消息。同时还要处理 Skills 系统提示的组装、历史上下文裁剪。这样既能实时显示，也能确保状态一致。再加上 localStorage 持久化和旧结构迁移，刷新后仍能恢复上下文，这部分是我认为最有技术含量的地方。

---

### Q2：为什么你要设计"多会话模型"，而不是一个消息数组？

短答版：
因为真实聊天产品一定有会话概念，一个消息数组无法支持会话切换、重命名、删除、排序这些能力。

展开版：
如果只有一个消息数组，后续做"历史会话"会非常痛苦。多会话模型下，每个会话是一个独立实体，包含 `id/title/messages/updatedAt`。这样新增功能都自然：

1. 会话切换只改 `currentConversationId`
2. 会话列表按 `updatedAt` 排序
3. 重命名和删除都在会话维度操作
4. 首条用户消息可自动生成会话标题

这个模型是从"可演进性"出发设计的，不是为了当前功能勉强实现。

---

### Q3：流式返回为什么体验更好？前端具体怎么处理？

短答版：
流式让用户更早看到结果，减少等待焦虑。前端通过 SSE 按 chunk 追加文本，并在结束标记 `[DONE]` 时收口状态。

展开版：
我在前端 `sendMessageStream` 里读取 `response.body.getReader()`，逐段解码后按行解析 `data:`。每个 chunk 到达都触发 `onChunk`，追加到当前 AI 占位消息。收到 `[DONE]` 或流结束时执行 `onDone`，把 `isLoading/currentStreamingId` 置回空闲。错误时走 `onError`，写入兜底文案并结束 loading。这个流程保证了 UI 不会卡在"正在思考"。

---

### Q4：Skills 功能是怎么设计和实现的？

短答版：
Skills 是一种自定义 AI 行为的机制，用户从 GitHub 导入 SKILL.md，系统在发消息时将其作为系统提示注入。

展开版：
设计上，Skills 是独立的状态域，存在 `skill.store.js` 里。导入时，用户粘贴 GitHub blob 链接，我做了 URL 转换逻辑，把 `github.com/.../blob/...` 转成 `raw.githubusercontent.com/...`，然后 fetch 内容。解析时提取 Markdown 标题作为 Skill 名称，正文作为指令。发送消息时，`buildSystemPrompt` 把启用的 Skills 组装成系统消息，插入到对话历史前面。这样 AI 就会按照用户导入的规则来回答。

---

### Q5：你如何保证聊天上下文质量？

短答版：
发送前做历史清洗和窗口裁剪，保留有效上下文，去掉噪声消息。

展开版：
我在 `buildHistory` 做了四件事：

1. 去掉欢迎消息和错误消息
2. 只取最近窗口（避免无限膨胀）
3. 连续同角色去重（减少上下文污染）
4. 避免以不合理角色结尾

本质是把"状态存储格式"转成"模型更容易理解的上下文格式"。

---

### Q6：为什么状态要放 Pinia，不放组件内部？

短答版：
因为这些状态跨页面、跨组件共享，而且需要刷新恢复，放组件会导致状态碎片化。

展开版：
比如聊天数据会被 `AIChat/ConversationList/MessageList` 同时消费，还要在路由切换后保留。组件状态适合临时 UI 状态，不适合业务主状态。放 Pinia 后，数据源唯一、调试清晰，也方便做持久化和后续测试。

---

### Q7：项目里做了哪些可维护性设计？

短答版：
做了分层架构、store 按领域拆分、API 统一封装、全局反馈组件复用。

展开版：
可维护性我主要做了这些：

1. `views` 只关心页面组合，`components` 关注可复用 UI 逻辑
2. `stores` 按业务域拆分，避免单 store 巨型化
3. `api` 层封装网络细节，页面不直接写 fetch
4. Toast 做成全局能力，避免重复实现
5. 路由守卫统一做权限入口，避免页面各自判断登录

---

### Q8：你做了哪些性能相关工作？

短答版：

## 19. 30 秒 / 60 秒 / 120 秒项目口述模板

### 30 秒版（电梯陈述）

我做了一个 Vue 3 的 AI 聊天助手，核心是多会话流式聊天。技术上用 Pinia 做状态分层，用 SSE 做增量渲染，还设计了 Skills 功能支持自定义 AI 行为。整体是可扩展的前端架构。

### 60 秒版（常规面试）

项目采用 Vue 3 + Pinia + Router + Tailwind。登录后进入聊天页面，左侧是会话列表，右侧是聊天区域。聊天是核心：我设计了会话级数据模型，支持新建、切换、重命名、删除；发送消息时先落本地状态，再走 `/api/stream` 进行 SSE 流式更新，用户能实时看到回复生成。还设计了 Skills 功能，支持从 GitHub 导入自定义提示词。为了保证稳定性，我做了错误兜底、上下文裁剪和持久化恢复。工程层面做了路由懒加载和 Vite 分包，兼顾体验与可维护性。

### 120 秒版（深挖开场）

这个项目我把它当一个"可演进产品"在做，而不是一次性 demo。架构上我先做分层：App 壳层负责统一布局，Router 负责权限入口，Pinia 负责领域状态，API 层负责请求封装。核心难点在聊天链路，我把数据设计成多会话模型，每个会话有独立消息和时间戳；发送时先把用户消息落库并创建 AI 占位消息，然后通过 SSE 按 chunk 追加文本，结束时再统一收口 loading 状态。这样既保证实时体验，也保证状态一致性。我还设计了 Skills 功能，让用户可以导入自定义提示词来增强 AI 回答。为了上线可用，我还做了本地持久化、旧版本数据迁移、错误兜底、语言和主题的全局联动。后续迭代重点是把 mock 鉴权和本地数据切到真实后端持久化，并补齐测试和监控，形成完整生产链路。

## 20. 临场应对技巧（简短）

1. 被问"你做了什么"时，先讲"我解决了什么问题"，再讲"我怎么做"。
2. 被问技术细节时，先给结论，再给 2-3 个关键实现点。
3. 被挑战"这不就是 demo 吗"时，主动说风险和改进路线，体现工程判断力。
4. 遇到不会的问题，不要硬答；说清假设和你会怎么验证。
