# 武理小精灵：项目流程详解

> 适用场景：前端面试复习、项目讲解、简历项目深挖

---

## 1. 先用一句话理解这个项目

这是一个 **AI 聊天助手** 应用。
你登录后，可以：

1. 和 AI 助手聊天（支持流式输出）
2. 管理多个聊天会话
3. 导入 GitHub Skills 自定义 AI 行为
4. 管理自定义提示词（新增、编辑、删除、选中应用）
5. 配置 MCP 服务器扩展 AI 能力
6. 管理知识库文档（RAG 检索增强生成）
7. 在线运行 JavaScript 代码
8. 切换深色/浅色主题、多语言支持
9. 查看用量统计（Token 消耗、成本估算）

前端负责"页面和交互"，后端负责"提供接口和 AI 服务"。

---

## 2. 项目里最重要的几个概念

### 2.1 前端是什么？

前端就是你在浏览器里看到的页面和操作逻辑。
本项目前端技术：

1. **Vue 3**：写页面组件，使用 Composition API
2. **Pinia**：管理全局状态（用户信息、聊天记录、Skills、MCP、知识库）
3. **Vue Router**：页面跳转（`/login`、`/chat`、`/knowledge`）
4. **Tailwind CSS**：快速写样式
5. **Vite**：开发和打包工具
6. **i18n**：国际化支持（中/英文切换）

### 2.2 后端是什么？

后端是运行在服务器上的程序，主要提供 API 接口。
本项目后端是 Node.js + Express，负责：

1. 登录接口
2. 聊天接口（普通 + 流式）
3. 对接讯飞 Spark 模型
4. RAG 知识库管理（文档存储、向量检索）
5. 向量嵌入服务（Embedding）
6. 会话持久化（Redis 可选）
7. 用量统计与成本追踪

### 2.3 "状态"是什么？

状态就是"程序当前记住的数据"。
比如：

1. 当前登录用户是谁
2. 当前是深色模式还是浅色模式
3. 当前聊天会话是哪一个
4. 每个会话有哪些消息
5. 已导入哪些 Skills
6. 已选中哪个自定义提示词
7. 已配置哪些 MCP 服务器
8. 当前使用的语言（中/英文）

这些状态放在 Pinia 里，页面只负责显示和触发动作。

## 4. 整个项目"从启动到可用"的完整流程

### 4.1 启动阶段

1. 运行前端：`npm run dev`
2. 运行后端：`cd backend && npm run dev`
3. 浏览器打开前端地址（默认 5173）

此时 Vite 会把前端里的 `/api` 请求转发到后端 `http://localhost:3000`。

### 4.2 前端初始化发生了什么？

入口文件：`src/main.js`

它做三件事：

1. 创建 Vue 应用
2. 挂载 Pinia（状态管理）
3. 挂载 Router（路由）

然后挂到页面根节点 `#app`。

### 4.3 全局壳层如何工作？

核心文件：`src/App.vue`

它负责：

1. 如果当前是 `/login`，只渲染登录页
2. 如果不是 `/login`，渲染完整后台布局：
   - 左侧 Sidebar（会话列表）
   - 顶部 Header（标题、用户头像）
   - 中间主内容区（路由页面）
3. 挂全局组件：`ToastManager`（全局提示）

你可以把它理解成"网站外壳"。

## 5. 登录流程

页面：`src/views/Login.vue`
状态：`src/stores/auth.store.js`
路由守卫：`src/router/index.js`

### 5.1 用户行为

1. 输入账号和密码
2. 点击登录按钮

### 5.2 前端逻辑

1. 先做空值校验
2. 模拟请求延迟
3. 密码为 `123456` 时视为成功
4. 调 `authStore.login(...)` 写入用户信息
5. 显示成功 toast
6. 跳转聊天页 `/chat`

### 5.3 路由守卫逻辑

1. 访问受保护页面且未登录 -> 跳到 `/login`
2. 已登录还访问 `/login` -> 重定向 `/chat`

这就是"前端权限门禁"。

5. 消息列表：`src/components/chat/MessageList.vue`
6. 输入框：`src/components/chat/ChatBox.vue`
7. 语音输入：`src/components/chat/VoiceRecorder.vue`
8. Markdown 渲染：`src/components/chat/MarkdownRenderer.vue`
9. Skills 面板：`src/components/chat/SkillPanel.vue`
10. 提示词面板：`src/components/chat/PromptPanel.vue`
11. MCP 面板：`src/components/chat/McpPanel.vue`
12. 代码运行器：`src/components/chat/CodeRunner.vue`
13. 功能面板：`src/components/chat/FeaturePanel.vue`
14. 性能面板：`src/components/chat/PerformancePanel.vue`

### 6.2 用户发送一条消息后发生什么？

#### 第 1 步：输入

用户在 `ChatBox` 输入文字，按 Enter 或点发送。

#### 第 2 步：触发 store 发送

`AIChat.vue` 调 `chatStore.sendMessage(message)`。

#### 第 3 步：store 先改本地状态

在 `chat.store.js` 里：

1. 先把用户消息加入当前会话
2. 创建一个空的 AI 消息占位（用于流式追加）
3. `isLoading = true`
4. 记录当前流式消息 id：`currentStreamingId`

这样 UI 能立刻反馈，不用等接口返回。

#### 第 4 步：构建历史上下文

`buildHistory(...)` 会做清洗：

1. 去掉 welcome 消息
2. 去掉错误消息
3. 取最近 20 条
4. 修正连续同角色问题

目的是提高模型上下文质量。

#### 第 5 步：构建系统提示（Skills + 提示词）

如果有启用的 Skills 或选中的提示词：

1. **提示词优先**：`promptStore.getPromptContent()` 获取选中的自定义提示词
2. **Skills 组装**：`skillStore.buildSystemPrompt()` 取最多 4 个启用的 Skill
3. **合并策略**：两者都存在时拼接，优先使用自定义提示词
4. 插入到对话历史前面作为系统提示

#### 第 6 步：调用流式接口

1. 通过 `sendMessageStream` 向 `/api/stream` 发起流式请求，采用 SSE（Server-Sent Events）风格进行数据传输。

   **核心实现要点**：

   1. **流式解析**：使用 `response.body.getReader()` 获取 `ReadableStream`，配合 `TextDecoder` 实现增量解码。
   2. **缓冲区设计**：针对网络分包导致的数据不完整问题，引入 `buffer` 变量暂存未完成的行，仅当接收到完整 `data:` 行时才进行 JSON 解析，有效避免解析异常。
   3. **渲染优化**：在 `onChunk` 回调中，使用 `requestAnimationFrame` 合并高频文本更新，降低 Vue 响应式系统的触发频率，确保长文本输出时的界面流畅度。
   4. **生命周期管理**：
      - `onChunk`：将增量文本追加至预先创建的 AI 占位消息
      - `[DONE]`：触发 `onDone`，终止 loading 状态并执行会话标题生成
      - `onError`：捕获异常，将占位消息标记为错误态并呈现用户友好的错误提示

#### 第 7 步：页面实时刷新

`MessageList.vue` 监听消息变化自动滚动到底部。
用户看到的效果就是"AI 一边生成一边显示"。

## 7. 会话管理机制（为什么这是亮点）

`chat.store.js` 不是"单数组消息"，而是"多会话模型"：

1. `conversations`：所有会话
2. `currentConversationId`：当前会话 id
3. `currentConversation`：当前会话对象（computed）
4. `messages`：当前会话消息列表（computed）

支持功能：

1. 新建会话
2. 切换会话
3. 重命名会话
4. 删除会话
5. 会话列表按更新时间排序
6. 首条用户消息自动生成标题

## 8. Skills 功能（新增亮点）

### 8.1 什么是 Skills？

Skills 是一种"自定义 AI 行为"的机制。用户可以从 GitHub 导入 SKILL.md 文件，让 AI 按照特定风格或规则回答问题。

### 8.2 相关文件

- `src/stores/skill.store.js`：Skills 状态管理
- `src/components/chat/SkillPanel.vue`：Skills 面板 UI

### 8.3 导入流程

1. 用户粘贴 GitHub blob 链接（如 `https://github.com/xxx/blob/main/SKILL.md`）
2. 前端将链接转换为 raw 链接（`https://raw.githubusercontent.com/...`）
3. 通过 fetch 获取 Markdown 内容
4. 解析标题和描述，存入 localStorage
5. 发送消息时，将启用的 Skills 组装成系统提示

### 8.4 技术细节

- **URL 转换**：支持 `github.com/blob/...` 和 `raw.githubusercontent.com/...` 两种格式
- **去重**：相同 raw URL 的 Skill 不会重复导入
- **限制**：最多 4 个 Skill 同时启用，每个 Skill 内容限制 6000 字符

## 9. MCP 服务器配置功能（新增亮点）

### 9.1 什么是 MCP？

MCP（Model Context Protocol）是一种让 AI 模型与外部工具交互的协议。通过配置 MCP 服务器，可以扩展 AI 的能力，例如访问本地文件、调用外部 API、操作数据库等。

### 9.2 相关文件

- `src/stores/mcp.store.js`：MCP 服务器状态管理
- `src/components/chat/McpPanel.vue`：MCP 配置面板 UI

### 9.3 功能特性

1. **添加服务器**：配置 MCP 服务器的名称、命令、参数、环境变量
2. **编辑服务器**：修改已有服务器配置
3. **删除服务器**：移除不需要的服务器
4. **复制服务器**：快速复制配置作为模板
5. **启用/禁用**：切换服务器是否生效
6. **导入配置**：支持 Claude Desktop 格式的 JSON 配置导入
7. **导出配置**：生成 Claude Desktop 兼容的配置文件

### 9.4 技术细节

- **配置结构**：每个服务器包含 `id`、`name`、`command`、`args`、`env`、`enabled`、`description`
- **本地存储**：配置自动保存到 localStorage（`mcp_servers`）
- **格式兼容**：支持导入导出 Claude Desktop 的 `mcpServers` 配置格式

```javascript
// Claude Desktop 配置格式示例
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/files"],
      "env": {}
    }
  }
}
```

## 10. RAG 知识库功能（新增亮点）

### 10.1 什么是 RAG？

RAG（Retrieval-Augmented Generation，检索增强生成）是一种结合外部知识库的 AI 增强技术。用户提问时，系统先从知识库检索相关文档，然后将检索结果作为上下文提供给 AI，从而生成更准确、更有针对性的回答。

### 10.2 相关文件

**前端：**
- `src/views/KnowledgeBase.vue`：知识库管理页面
- `src/api/rag.js`：RAG API 封装

**后端：**
- `backend/src/services/rag.service.js`：RAG 核心服务
- `backend/src/services/embedding.service.js`：向量嵌入服务
- `backend/src/services/chroma.service.js`：Chroma 向量数据库服务
- `backend/src/services/document.service.js`：文档管理服务
- `backend/src/services/file-upload.service.js`：文件上传服务
- `backend/src/routes/rag.routes.js`：RAG 路由

### 10.3 功能特性

1. **文档管理**：添加、删除、查看知识库文档
2. **文件上传**：支持 PDF、DOCX、TXT、MD 格式文件上传
3. **分类管理**：按类别组织文档（学校概况、计算机学院、图书馆、教务相关等）
4. **向量检索**：基于语义相似度检索相关文档
5. **统计信息**：查看文档数量、向量数量等统计

### 10.4 技术架构

```
用户提问 → Embedding 向量化 → Chroma 向量检索 → 构建上下文 → LLM 生成回答
```

**核心流程：**

1. **文档入库**：
   - 用户上传文档或输入文本
   - 调用讯飞 Embedding API 生成向量
   - 存储到 Chroma 向量数据库

2. **检索增强**：
   - 用户提问时，先将问题转为向量
   - 在 Chroma 中进行相似度搜索（Top-K）
   - 将检索到的文档作为上下文注入提示词

3. **流式响应**：
   - 支持 RAG 增强的流式对话
   - 先返回检索到的来源信息
   - 再流式输出 AI 回答

### 10.5 API 接口

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/api/rag/documents` | 获取文档列表 |
| POST | `/api/rag/documents` | 添加文档 |
| POST | `/api/rag/documents/upload` | 上传文件 |
| POST | `/api/rag/documents/batch` | 批量添加文档 |
| DELETE | `/api/rag/documents/:id` | 删除文档 |
| GET | `/api/rag/stats` | 获取统计信息 |

## 11. 代码运行器功能（新增亮点）

### 11.1 功能概述

代码运行器允许用户在浏览器中直接运行 JavaScript 代码，实时查看输出结果，无需配置本地开发环境。

### 11.2 相关文件

- `src/components/chat/CodeRunner.vue`：代码运行器组件

### 11.3 技术实现

1. **沙箱隔离**：使用 iframe 创建隔离的执行环境
2. **Console 拦截**：重写 `console.log/error/warn/info` 方法，捕获输出
3. **错误捕获**：通过 `window.onerror` 捕获运行时错误
4. **执行计时**：使用 `performance.now()` 记录执行时间
5. **消息通信**：通过 `postMessage` 实现父子窗口通信

### 11.4 安全考虑

- 代码在 iframe 沙箱中执行，与主页面隔离
- 仅支持 JavaScript 语言
- 无法访问主页面的 DOM 和敏感 API

## 12. 提示词管理功能

### 12.1 什么是提示词管理？

提示词管理是一种"自定义 AI 行为"的机制。用户可以创建、编辑、删除自定义提示词，选中后作为系统提示发送给 AI，让 AI 按照特定风格或规则回答问题。

### 12.2 相关文件

- `src/stores/prompt.store.js`：提示词状态管理
- `src/components/chat/PromptPanel.vue`：提示词面板 UI

### 12.3 功能特性

1. **新建提示词**：点击"新建"按钮添加自定义提示词
2. **编辑提示词**：修改已有提示词的名称、分类、内容
3. **删除提示词**：移除不需要的提示词
4. **复制提示词**：快速复制现有提示词作为模板
5. **分类筛选**：按分类过滤提示词列表
6. **选中应用**：点击提示词卡片选中/取消，选中的提示词会在对话中生效
7. **默认模板**：内置"代码助手"、"翻译专家"、"学习助手"三个默认提示词

### 12.4 技术细节

- **本地存储**：提示词自动保存到 localStorage（`custom_prompts`）
- **与 Skills 协同**：提示词和 Skills 可以同时生效，两者拼接后作为系统提示
- **优先级**：自定义提示词优先于 Skills

## 13. 本地持久化机制

本项目把核心状态存进了 localStorage：

1. 聊天会话：`chat_conversations`
2. 当前会话 id：`chat_current_conversation_id`
3. Skills：`chat_skills`
4. 自定义提示词：`custom_prompts`
5. MCP 服务器配置：`mcp_servers`
6. 主题：`darkMode`
7. 语言设置：`language`

另外还做了旧数据迁移：

1. 检测旧 key `chat_messages`
2. 自动转换成新会话结构
3. 删除旧 key

这体现了"版本升级兼容意识"。

`POST /api/stream` 接口：

**请求体：**

**响应格式：** Server-Sent Events (SSE)

```
data: {"content": "你"}
```

**技术要点：**
1. 设置响应头：`Content-Type: text/event-stream`

3. 调用 `xunfeiService.getCompletionStream()` 获取异步迭代器
4. 逐 chunk 写入响应

### Q2：为什么你要设计"多会话模型"，而不是一个消息数组？

短答版：
因为真实聊天产品一定有会话概念，一个消息数组无法支持会话切换、重命名、删除、排序这些能力。

展开版：
如果只有一个消息数组，后续做"历史会话"会非常痛苦。多会话模型下，每个会话是一个独立实体，包含 `id/title/messages/updatedAt`。这样新增功能都自然：

1. 会话切换只改 `currentConversationId`
2. 会话列表按 `updatedAt` 排序
3. 重命名和删除都在会话维度操作
4. 首条用户消息可自动生成会话标题

### Q3：流式返回为什么体验更好？前端具体怎么处理？

短答版：
流式让用户更早看到结果，减少等待焦虑。前端通过 SSE 按 chunk 追加文本，并在结束标记 `[DONE]` 时收口状态。

展开版：
我在前端 `sendMessageStream` 里读取 `response.body.getReader()`，逐段解码后按行解析 `data:`。每个 chunk 到达都触发 `onChunk`，追加到当前 AI 占位消息。收到 `[DONE]` 或流结束时执行 `onDone`，把 `isLoading/currentStreamingId` 置回空闲。错误时走 `onError`，写入兜底文案并结束 loading。这个流程保证了 UI 不会卡在"正在思考"。

## 29. 消息滚动与锚点跳动问题（场景题重点）

### 29.1 问题本质

- 消息内容由模型流式返回，高度动态变化
- 浏览历史消息时，如果新消息/高度变化，会导致 scrollTop 突变（锚点跳动）

### 29.2 常见问题来源

- DOM 渲染后高度变化，浏览器自动重排
- 在流式更新（SSE/stream）中不断 append 内容
- 没有区分用户是否在底部

### 29.3 处理思路

#### ① 引入"锚点"概念

- 在消息列表底部维护一个 anchorOffset
- 更新前记录：`const prevScrollHeight = container.scrollHeight`
- 更新后：`container.scrollTop += container.scrollHeight - prevScrollHeight`

#### ② 区分"自动滚动"和"用户滚动"

- 如果用户在底部（阈值判断，如 50px 内）→ 自动跟随
- 如果用户在历史区域 → 锁住 scrollTop，不自动滚

#### ③ 对流式渲染做节流

- SSE 每 token 都 setState → 极易抖动
- 实际做法：
  - requestAnimationFrame / debounce
  - 或按字符数/时间批量 flush

### 29.4 代码实现

```javascript
const handleScroll = () => {
  const container = scrollContainer.value
  if (!container) return

  const { scrollTop, scrollHeight, clientHeight } = container
  // 判断是否在底部（50px 阈值）
  const isNearBottom = scrollHeight - scrollTop - clientHeight < 50

  if (isNearBottom && shouldAutoScroll.value) {
    // 自动滚动到底部
    nextTick(() => {
      container.scrollTop = container.scrollHeight
    })
  }
}

// 流式更新时的处理
const appendChunk = (chunk) => {
  const prevHeight = container.scrollHeight
  currentMessage.value += chunk
  nextTick(() => {
    if (isNearBottom()) {
      container.scrollTop = container.scrollHeight
    } else {
      // 保持相对位置
      container.scrollTop += container.scrollHeight - prevHeight
    }
  })
}
```

## 31. SSE 重连后重复消息问题（高频考点）

### 31.1 可能原因（重点）

1. 后端未提供 messageId/cursor
2. 前端重连后从 0 开始 append
3. Abort 未生效，旧连接仍在推数据

### 31.2 验证方式

- Network 查看是否存在多个 EventSource
- log messageId/index 是否重复
- 断网 → 重连 → 对比 message 序列

### 31.3 解决方案

**前端：**
- 用 lastMessageId
- Set/Map 去重

**协议级：**
- reconnect 传 cursor

**生命周期：**
- 严格 abort 旧连接

### 31.4 代码实现

```javascript
const eventSource = ref(null)
const lastMessageId = ref('')
const receivedIds = new Set()

const connect = () => {
  // 先关闭旧连接
  if (eventSource.value) {
    eventSource.value.close()
  }

  const url = lastMessageId.value
    ? `/api/stream?last_id=${lastMessageId.value}`
    : '/api/stream'

  eventSource.value = new EventSource(url)

  eventSource.value.onmessage = (e) => {
    const data = JSON.parse(e.data)

    // 去重检查
    if (receivedIds.has(data.id)) return
    receivedIds.add(data.id)

    lastMessageId.value = data.id
    // 处理消息...
  }

  eventSource.value.onerror = () => {
    // 指数退避重连
    setTimeout(connect, getRetryDelay())
  }
}
```

## 33. 接口不稳定与网络异常处理

### 33.1 瞬时故障处理：自动重试与退避

主要应对短暂的网络波动、临时的 API 服务拥塞（如 HTTP 503 或 429 错误）。

- **识别可重试错误**：仅对 HTTP 5xx 错误、429 限流错误以及网络 I/O 异常进行处理。
- **指数退避重试（Exponential Backoff）**：我们不会立即重试，而是采用指数级增长的等待间隔。例如，第一次失败等待 1s，第二次 4s，第三次 16s。
- **随机抖动（Jitter）**：在计算出的等待时间上随机增加一个短时间，避免所有客户端在同一时刻重试，防止对 API 造成"惊群效应"。
- **最大限制**：设置最大重试次数（如 5 次）和最大等待时间，防止无限重试耗尽资源。

### 33.2 流式数据（SSE/流 API）中断恢复

针对 AI 题目生成这种流式输出中途断开的特殊情况。

- **前端断点捕获**：前端 JavaScript 监听流的 close 或 error 事件，并精确记录已接收和显示的所有内容。
- **续写 Prompt 机制**：当连接断开后，重试发出的请求会带上已生成的完整文本作为新的上下文（Prompt）。同时，附带明确的指令："请紧接在以上内容之后，继续完成你的生成任务，不要重复。"
- **SSE 自动重连**：如果使用的是 SSE，浏览器内置的 EventSource 会自动尝试重连，并在请求头中发送 Last-Event-ID，后端则根据此 ID 从断点处恢复数据传输。

### 33.3 长期故障与资源管理

主要应对 API 长期不稳定、用户中途取消，以及保护系统资源的情况。

- **熔断器（Circuit Breaker）**：如果某个 API 在短时间内连续失败多次，系统会启动熔断器，暂时停止向该 API 发送请求一段时间，直接返回错误，避免重复失败和资源浪费。
- **级联取消（Cancellation Propagation）**：当用户中途关闭页面或取消操作时，我们的后端会立即检测到 HTTP 连接断开，并向大模型 API 发送取消信号，要求其立即停止计算。这能有效节省 API 计费成本。
- **服务降级**：如果核心 API 长期不可用，我们可以启动降级策略，例如：
  - 切换到备用大模型 API 供应商。
  - 暂时提供缓存的、预设的题目列表，确保核心功能（获取题目）仍能使用。

### 33.4 代码实现

```javascript
class ResilientAPIClient {
  constructor() {
    this.retryLimit = 5
    this.baseDelay = 1000
    this.circuitBreaker = {
      failures: 0,
      threshold: 3,
      isOpen: false,
      resetTime: 30000
    }
  }

  async request(config, retryCount = 0) {
    // 熔断器检查
    if (this.circuitBreaker.isOpen) {
      throw new Error('Circuit breaker is open')
    }

    try {
      const response = await fetch(config)
      this.resetCircuitBreaker()
      return response
    } catch (error) {
      this.circuitBreaker.failures++

      if (this.circuitBreaker.failures >= this.circuitBreaker.threshold) {
        this.openCircuitBreaker()
      }

      if (this.shouldRetry(error) && retryCount < this.retryLimit) {
        const delay = this.getRetryDelay(retryCount)
        await this.sleep(delay)
        return this.request(config, retryCount + 1)
      }

      throw error
    }
  }

  getRetryDelay(retryCount) {
    const baseDelay = this.baseDelay * Math.pow(2, retryCount)
    const jitter = Math.random() * 1000 // 随机抖动
    return baseDelay + jitter
  }

  shouldRetry(error) {
    return error.status >= 500 || error.status === 429
  }
}
```

### 34.1 问题背景

### 34.2 优化方案

#### ① RAG（检索增强生成）

它将所有历史对话转化为向量存储在外部数据库中，在用户提问时，仅检索并提交与新问题语义上最相关的几条历史记录作为上下文，从而实现高效的长期记忆。

#### ② 滑动窗口

滑动窗口是最简单的方案，它通过设定一个固定的轮次或 Token 数量，在每次提交给大模型时，只保留最近 x 轮对话，从而直接控制输入数据的量级，但可能会丢失窗口之外的关键信息。

```javascript
const buildContext = (messages, windowSize = 20) => {
  return messages.slice(-windowSize)
}
```

累进式摘要利用大模型的能力，将旧的对话内容压缩成一段精简的记忆摘要，每次只提交旧摘要和最新几轮对话，以在不丢失关键信息的前提下，有效削减 Token 数量。

## 35. 虚拟列表渲染判断原理

### 35.1 固定高度列表

判断一个列表项是否需要渲染，主要依赖于监听 onScroll 事件后获取的 scrollTop、clientHeight 和预设的 itemHeight 三个值；通过将 scrollTop 除以 itemHeight 来计算出当前视口内的起始索引（startIndex），并结合 clientHeight 确定结束索引（endIndex）；为了防止快速滚动时出现白屏，必须在计算出的可视范围上下额外增加一个冗余缓冲区，只有位于这个扩展渲染范围内的项目才会被实际保留在 DOM 中。

```javascript
const calculateVisibleRange = (scrollTop, clientHeight, itemHeight, bufferSize = 3) => {
  const startIndex = Math.max(0, Math.floor(scrollTop / itemHeight) - bufferSize)
  const endIndex = Math.ceil((scrollTop + clientHeight) / itemHeight) + bufferSize
  return { startIndex, endIndex }
}
```

### 35.2 不定高列表

在不定高列表中，由于每个列表项的高度未知且不同，无法通过简单除法计算起始索引，因此，必须在滚动时维护一个所有列表项的累积高度映射表，然后通过二分查找或其他高效查找算法，找到第一个累积高度（大于或等于当前 scrollTop 的元素索引（模拟的 offsetTop）大于或等于当前 scrollTop 的列表项索引），就是在不定高场景下的实现原理，确保了渲染起始位置的精确判断。

```javascript
class DynamicHeightList {
  constructor() {
    this.heightMap = new Map() // id -> height
    this.positionCache = [] // 累积位置缓存
  }

  // 二分查找起始索引
  findStartIndex(scrollTop) {
    let left = 0, right = this.positionCache.length - 1
    while (left < right) {
      const mid = Math.floor((left + right) / 2)
      if (this.positionCache[mid] < scrollTop) {
        left = mid + 1
      } else {
        right = mid
      }
    }
    return left
  }

  // 更新高度缓存
  updateHeight(id, height) {
    this.heightMap.set(id, height)
    this.rebuildPositionCache()
  }
}
```

## 36. SSE 自动重连实现

### 36.1 SSE 内置机制

SSE 本身内置了自动重连机制：
- 网络断开、服务端中断、服务端返回非 200 状态码（短暂）由浏览器自动处理
- 服务器可以通过 `retry:` 字段控制重连间隔
- `id:` + `Last-Event-ID` 机制保证断点续传
- 前端可以在 onerror 中补充策略

### 36.2 完整实现

```javascript
class SSEClient {
  constructor(url, options = {}) {
    this.url = url
    this.options = options
    this.eventSource = null
    this.retryCount = 0
    this.maxRetry = options.maxRetry || 5
    this.lastEventId = ''
  }

  connect() {
    if (this.eventSource) {
      this.eventSource.close()
    }

    const url = this.lastEventId
      ? `${this.url}?lastEventId=${this.lastEventId}`
      : this.url

    this.eventSource = new EventSource(url)

    this.eventSource.onmessage = (e) => {
      this.retryCount = 0 // 成功后重置重试计数
      const data = JSON.parse(e.data)
      if (data.id) this.lastEventId = data.id
      this.options.onMessage?.(data)
    }

    this.eventSource.onerror = (e) => {
      console.error('SSE error:', e)

      if (this.eventSource.readyState === EventSource.CLOSED) {
        this.retry()
      }
    }
  }

  retry() {
    if (this.retryCount >= this.maxRetry) {
      this.options.onError?.(new Error('Max retry exceeded'))
      return
    }

    this.retryCount++
    const delay = Math.min(1000 * Math.pow(2, this.retryCount), 30000)

    setTimeout(() => this.connect(), delay)
  }

  disconnect() {
    if (this.eventSource) {
      this.eventSource.close()
      this.eventSource = null
    }
  }
}
```

|------|------|-----|-----------|
| 通信方向 | 单向（客户端请求） | 单向（服务端推送） | 双向 |

4. **资源效率**：相比 WebSocket 更轻量，减少协议开销

## 41. Axios 二次封装实现方案

### 41.1 基础封装：创建实例与类型定义

```typescript
import axios, { AxiosInstance, AxiosRequestConfig, AxiosResponse, AxiosError } from 'axios';

type RequestConfig<T = any> = AxiosRequestConfig & {
  retry?: number; // 自定义重试次数
  skipErrorHandler?: boolean; // 跳过统一错误处理
};

type ResponseData<T = any> = {
  code: number;
  message: string;
  data: T;
};

class Request {
  private instance: AxiosInstance;
  private retryLimit = 3; // 默认重试次数
  private isRefreshing = false; // Token 刷新状态锁
  private retryQueue: Array<() => void> = []; // 重试队列

  constructor(baseURL: string) {
    this.instance = axios.create({
      baseURL,
      timeout: 10000,
      headers: { 'Content-Type': 'application/json' }
    });
    this.initInterceptors();
  }

  private initInterceptors() {
    // 请求拦截器
    this.instance.interceptors.request.use(
      (config: RequestConfig) => {
        const token = localStorage.getItem('token');
        if (token) {
          config.headers.Authorization = `Bearer ${token}`;
        }
        return config;
      },
      (error: AxiosError) => Promise.reject(error)
    );

    // 响应拦截器
    this.instance.interceptors.response.use(
      (response: AxiosResponse<ResponseData>) => {
        const { data } = response;
        if (data.code !== 200 && !config.skipErrorHandler) {
          return Promise.reject(data);
        }
        return data;
      },
      async (error: AxiosError) => {
        const originalRequest = error.config as RequestConfig;

        // 网络错误处理
        if (!error.response) {
          return this.handleNetworkError(originalRequest);
        }
        // 授权过期处理
        if (error.response.status === 401) {
          return this.handleUnauthorized(originalRequest);
        }
        // 服务器错误处理
        return this.handleServerError(error, originalRequest);
      }
    );
  }

  public request<T = any>(config: RequestConfig): Promise<T> {
    return this.instance.request(config);
  }
}
```

### 41.2 Token 自动刷新机制

采用状态锁（isRefreshing）与请求队列（retryQueue）双重保障机制。当首次出现 401 错误时：
- 立即锁定刷新状态
- 调用 refreshToken() 获取新凭证
- 更新本地存储的 Token
- 批量重试队列中缓存的请求
- 最终解除锁定并清空队列

```javascript
private async handleUnauthorized(originalRequest: RequestConfig) {
  if (!this.isRefreshing) {
    this.isRefreshing = true;
    try {
      const newToken = await refreshToken();
      localStorage.setItem('token', newToken);
      this.retryQueue.forEach(cb => cb());
      return this.instance(originalRequest);
    } catch (e) {
      localStorage.removeItem('token');
      window.location.href = '/login';
      return Promise.reject(e);
    } finally {
      this.isRefreshing = false;
      this.retryQueue = [];
    }
  }
  return new Promise((resolve) => {
    this.retryQueue.push(() => resolve(this.instance(originalRequest)));
  });
}
```

### 41.3 智能重试机制

针对网络波动设计指数退避重试策略：
- 基础延迟从 1 秒开始（1000ms × 2^重试次数）
- 每次重试延迟时间翻倍（1s → 2s → 4s）
- 最大重试次数由实例级 retryLimit 控制

```javascript
private handleNetworkError(originalRequest: RequestConfig) {
  if (originalRequest.retry ?? 0 < this.retryLimit) {
    const delay = 1000 * Math.pow(2, originalRequest.retry || 0);
    originalRequest.retry = (originalRequest.retry || 0) + 1;

    return new Promise(resolve =>
      setTimeout(() => resolve(this.instance(originalRequest)), delay)
    );
  }
  return Promise.reject(new Error('Network Error'));
}
```

## 42. 打字机效果频率计算与实现

### 42.1 核心思路

1. **后端分片输出**
   - LLM 模型按 token 或片段生成文本
   - 后端通过 SSE 每次发送一小段文本（chunk）给前端
   - 片段大小决定每次渲染的粒度

2. **前端渲染频率**
   - 假设用户感知最佳效果为每 50~100ms 更新一次屏幕
   - 如果每条 SSE 消息过小（例如每个字符），会导致频繁 DOM 更新 → 性能浪费
   - 如果每条 SSE 消息过大（例如 50 token 一次发送），用户看不到逐步生成的效果

### 42.2 前端实现

```javascript
import { ref } from 'vue'

const currentMessage = ref('')
let buffer = ''
let timer = null
const interval = 50 // 每 50ms 更新一次 DOM

function appendChunk(text) {
  buffer += text
  if (!timer) {
    timer = setTimeout(() => {
      currentMessage.value += buffer
      buffer = ''
      timer = null
    }, interval)
  }
}
```

**原理说明：**
- buffer 用来临时存储接收到的 chunk
- timer 确保每隔 interval 批量更新一次 DOM
- 保证即使 SSE 发送过快，也不会频繁重绘

### 42.3 进一步优化

1. **自适应节流**：根据消息长度和系统性能动态调整更新频率
2. **分片策略**：每个 chunk 可以按 token、单词或句子切分
3. **滚动同步**：使用 nextTick 保证用户看到最新文本

## 44. 多会话管理数据结构选择

### 44.1 方案对比

#### 数组管理会话

```javascript
const sessions = [
  { id: 's1', messages: [...] },
  { id: 's2', messages: [...] }
]
```

- 优点：结构直观，易于遍历
- 缺点：查找特定会话需要 O(n) 时间；删除或更新操作复杂

#### 对象字典管理

```javascript
const sessions = {
  s1: { messages: [...] },
  s2: { messages: [...] }
}
```

- 优点：按键查找会话 O(1)
- 缺点：无法保证插入顺序，迭代操作不如 Map 灵活

#### Map 管理会话（推荐方案）

```javascript
const sessions = new Map()

// 添加会话
sessions.set('s1', { messages: [] })
sessions.set('s2', { messages: [] })

// 获取会话
const session1 = sessions.get('s1')

// 更新会话消息
session1.messages.push({ role: 'user', content: '你好' })

// 删除会话
sessions.delete('s2')

// 遍历所有会话
sessions.forEach((value, key) => {
  console.log(key, value.messages.length)
})
```

### 44.2 Map 方案优势

1. **高效查找和更新**：增删改查单个会话 O(1)，比数组高效
2. **保留插入顺序**：Map 保持键值对插入顺序，可按创建顺序渲染会话列表
3. **灵活迭代**：提供 forEach、keys()、values()、entries() 方法
4. **易于与响应式状态结合**：在 Vue3 + Pinia 中封装 Map，使会话状态响应式更新

1. 先背第 1、4、6、7 节（主链路）
2. 再背第 8、9、10、12 节（体验和细节）
3. 然后背第 23-41 节（深度技术问答）
4. 最后背第 16、17、18 节（面试表达）

如果你能把"第 6 节聊天全链路"和"第 25-35 节技术难点"讲清楚，面试官会认为你不是只会写页面，而是理解完整业务系统和底层原理。
